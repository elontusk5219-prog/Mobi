# 行为上报与画像输入

**用途：** 定义 Room 内用户行为（戳击、长按、拖拽、沉默等）如何上报、以何种格式与通道进入 EverMemOS 或画像服务，以及画像侧如何消费与推断。与行为模块的接口约定：谁在何时触发上报、报文格式、画像侧推断规则。

**维护：** 与 [Mobi用户画像与进化驱动设计](Mobi用户画像与进化驱动设计.md)、[画像服务设计](画像服务设计.md)、[Mobi行为模块实现说明](Mobi行为模块实现说明.md) 同步。

---

## 1. 概述

- **目标**：Room 内的交互行为参与用户画像（人格维度与置信度），与对话文本一起被画像服务消费。
- **数据流**：客户端在发生行为时上报事件 → 写入 EverMemOS 或专用行为接口 → 画像服务读取/订阅 → 参与维度估计与置信度更新（见 [画像服务设计](画像服务设计.md) §2.3）。
- **行为模块职责**：行为模块负责「表现」（戳击动画、seeking 指令等）；**上报**可由同一视图/ViewModel 在触发行为时顺带调用，或由统一的上报服务集中处理。本契约约定「何时、以何格式、发往何处」。

---

## 2. 事件类型

| 类型 | 说明 | 触发时机 | 与画像推断 |
|------|------|----------|------------|
| **poke** | 戳击 Mobi | 每次 onPoke 回调（ProceduralMobiView） | 戳击频率 → 亲密倾向、宜人性、互动意愿 |
| **drag** | 拖拽 Mobi | 拖拽结束或按间隔采样（如每 2s 有位移则记一次） | 拖拽频次/时长 → 玩闹倾向、开放性 |
| **long_press** | 长按抚摸 | 长按结束（若已实现） | 长按时长 → 亲密/依赖程度 |
| **silence_interval** | 持续沉默 | 沉默超过阈值（如 15s）时上报一次，可带 duration_seconds | 沉默占比、谁先开口 → 外向性、依赖感 |
| **session_start** | 进入 Room | Room onAppear | 会话频次参与置信度衰减计算 |
| **session_end** | 离开 Room | Room onDisappear 或 app 退后台 | 同上 |

**谁先开口**：可由画像服务从 EverMemOS 对话顺序推断（首条为 user 或 assistant），无需客户端单独上报。

---

## 3. 通道方案

### 3.1 方案 A：复用 EverMemOS 存储（推荐 MVP）

- **方式**：使用现有 [EverMemOSClient.storeMemory](Mobi/Services/Network/EverMemOSClient.swift)，将行为事件写成一条「记忆」。
- **约定**：
  - `sender`：固定为 `"mobi_behavior"`（或 `"system"`），便于画像服务过滤。
  - `content`：JSON 字符串，见 §4。
  - `message_id`：唯一，如 `"behavior_\(userId)_\(eventType)_\(timestampMillis)"`。
  - `create_time`：ISO8601 或与现有 memories 一致格式。
  - `role`：可选 `"behavior"`；若 EverMemOS 支持按 type 过滤，可后续扩展。
- **画像消费**：画像服务在拉取该用户 memories 时，按 `sender == "mobi_behavior"` 或解析 `content` 中 `event_type` 识别行为事件，与对话类记忆分开处理；仅用行为事件更新维度/置信度，不参与 memoryContext 检索。

### 3.2 方案 B：独立行为接口（可选）

- **方式**：后端提供 `POST /api/v1/behavior` 或 `POST /profile/events`，请求体为事件数组。
- **适用**：希望行为与对话存储完全分离、或需要高频率上报且不愿污染 memories 时。
- **格式**：见 §4，单条或数组；需带 `user_id`（或由 Token 解析）。

---

## 4. 报文格式（单条事件）

**content JSON（方案 A 下 EverMemOS 的 content 字段）：**

```json
{
  "type": "behavior",
  "event": "poke",
  "ts": 1707890123456,
  "session_id": "optional-uuid"
}
```

**字段说明：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| type | string | 是 | 固定 `"behavior"`，便于画像解析 |
| event | string | 是 | `poke` \| `drag` \| `long_press` \| `silence_interval` \| `session_start` \| `session_end` |
| ts | number | 是 | 客户端时间戳（毫秒），用于排序与衰减 |
| session_id | string | 否 | 当次 Room 会话 ID，便于去重与按会话聚合 |
| duration_seconds | number | 否 | 仅 `silence_interval`：沉默持续秒数 |
| payload | object | 否 | 扩展用，如 drag 的位移量、long_press 时长等 |

**EverMemOS store 请求示例（方案 A）：**

```json
{
  "message_id": "behavior_usr1_poke_1707890123456",
  "create_time": "2024-02-14T12:35:23.456Z",
  "sender": "mobi_behavior",
  "content": "{\"type\":\"behavior\",\"event\":\"poke\",\"ts\":1707890123456}"
}
```

## 5. 与行为模块的接口约定

- **调用方**：Room 的 View 或 ViewModel（如 [RoomContainerView](Mobi/Features/Room/Views/RoomContainerView.swift)），在已有行为回调处增加「上报」调用；或引入 `BehaviorReportingService`，由 View 在 onPoke、拖拽结束、silence 超阈值、onAppear/onDisappear 时调用。
- **接口形态**（客户端内部）：
  - 推荐：统一入口如 `BehaviorReportingService.shared.record(event: .poke)`、`.record(event: .silenceInterval(durationSeconds: 15))` 等，内部负责组报文、调 EverMemOS store（或 POST 行为接口）。
  - 行为模块**不直接依赖**画像或 EverMemOS 实现细节，仅依赖「记录行为」的抽象（protocol 或 closure），便于测试与 Mock。
- **何时上报**：
  - **poke**：每次 onPoke 触发后调用一次；可做轻量节流（如 1s 内最多 1 条）避免连戳刷屏。
  - **drag**：拖拽结束时（gesture 的 .ended）上报一次，或每持续 N 秒上报一次。
  - **silence_interval**：由 tick 或定时器判断，当 `silenceDuration >= 15` 且本 session 未报过该段沉默时上报一条，带 `duration_seconds`。
  - **session_start / session_end**：RoomContainerView.onAppear 时 record(.sessionStart)，onDisappear 时 record(.sessionEnd)。

---

## 6. 画像侧推断规则（概要）

画像服务在消费行为事件时，可参考以下映射（具体权重与公式由实现决定）：

| 行为信号 | 可参与维度 | 方向 |
|----------|------------|------|
| 戳击频率高、长按多 | 宜人性、亲密倾向 | 正向 |
| 拖拽多、玩闹式交互 | 开放性、外向性 | 正向 |
| 沉默占比高、常由 Mobi 先开口 | 外向性 | 偏低；依赖感可升 |
| 会话频次、最后活动时间 | 置信度衰减 | last_activity 越久置信度衰减 |

详见 [画像服务设计](画像服务设计.md) §3、§6；行为事件与对话记忆一起参与 `computeDimensions` 与 `last_activity_at`。

---

## 7. 实现要点小结

- **客户端**：在 Room 内增加行为上报调用点（onPoke、拖拽结束、沉默阈值、onAppear/onDisappear）；报文符合 §4；方案 A 下使用 `EverMemOSClient.storeMemory`，sender=`"mobi_behavior"`，content=JSON 字符串。
- **画像服务**：拉取 memories 时区分 `sender == "mobi_behavior"`，解析 content 得到事件流；与 SoulProfile、对话记忆一起输入维度估计与置信度更新；`last_activity_at` 取最近一条 memory（含行为）的时间，用于置信度衰减。
- **行为模块**：无需改动现有表现逻辑；仅在与 View 的同一层增加对「记录行为」的调用，接口可抽象为 `record(event:)` 由上报服务实现。
