# 故事与日常节律：设计与施工表

**用途：** 故事驱动的自我学习与主动交互、Day 1 破冰、日常节律（睡—醒—玩、早晚仪式）的设计与实现施工表。供 Multi-Agent 严格按表逐项推进，确保上下文切换后无失忆。

**维护：** 与 [当前玩法说明-基于现有实现](当前玩法说明-基于现有实现.md)、[Mobi交互行为完整设计](Mobi交互行为完整设计.md) 同步。实现变更后必须同步本文档「施工表」状态。

---

## 一、设计概览

### 1.1 核心目标

- **消除面面相觑**：Mobi 先「闯祸」或「求助」，用户自然反应，而非用户想「该说什么」。
- **同居感而非聊天感**：用「生理节律（睡—醒—玩）」和「事件（做梦—发现—画画）」驱动，而非纯 Q&A。
- **强化 Mobi 身份**：通过故事、铭印、日常仪式，让用户意识到在用 Mobi 而非豆包。

### 1.2 三大模块

| 模块 | 说明 | 依赖 |
|------|------|------|
| **Part 0: Day 1 破冰** | **每次冷启动**先 AuthView 选择/注册 ID，再进 Room。初次进 Room：**Kuro 开场介绍（welcomeIntro 三屏 + 内嵌麦克风权限）**→ 落地→躲藏→探出→第一句问话→命名仪式；**学说话阶梯**：mute→gibberish→教会→铭印庆祝 | Room、Doubao |
| **Part 1: 故事机制** | 困惑→求助→用户回答→铭印；沉默/情绪/随机触发 | Room tick、onUserUtterance |
| **Part 2: 日常节律** | 早晨唤醒+梦境、白天平行陪伴、晚上日记+晚安 | 时间驱动、状态机 |

## 二、Part 0: Day 1 破冰

### 2.1 流程

1. **落地与迷失**：Mobi 掉进来、趴地、晕头转向、环顾、看到用户后躲到箱子后、露一只眼。
2. **第一次接触**：用户戳或说「别怕」→ Mobi 探出头、怯生生问「你是刚才那个声音吗？这里是哪里？我身体为什么变重了？」
3. **命名仪式**：Mobi 说「我不记得我叫什么了…Mobi？那是个好名字吗？」→ 用户确认 → Mobi 打滚、变项链「好，那我是 Mobi，你是 Star。」

### 2.2 与现有实现衔接

| 设计 | 现有 | 需新增 |
|------|------|--------|
| 落地 | mobiDropOffset 动画 | 已有，可复用 |
| 躲藏→探出 | 无 | 新增 Mobi 视觉状态：hiding / peeking |
| 第一句问话 | Wake-Up Trigger 通用开场 | 破冰专用：Day1 且 newborn 时发破冰困惑 |
| Kuro 开场介绍 | welcomeIntro 三屏 | 替代 guardian 弹窗；内嵌麦克风权限；选「稍后」可触摸但无语音 |
| 命名仪式 | AuthView 有用户名 | 新增：确认 Mobi 名字、Star 称呼；可选项链视觉 |

### 2.3 数据与状态

- `hasCompletedDay1IceBreaker: Bool`（按 userId 持久化）— 是否已完成破冰。
- `userConfirmedAsStar: Bool`（按 userId）— 用户是否已确认为 Star；可与 Auth 用户名关联。

## 三、Part 1: 故事机制（简要）

- **故事结构**：id, stage, triggerType(silence/emotion/random), confusionText, imprintTemplate, cooldownMinutes。
- **触发**：Room tick 沉默时故事优先于 seeking；onUserUtterance 情绪关键词触发 emotion 类故事。
- **铭印**：用户回答后识别并存储（ImprintService）；日记可展示「Star 教会了我…」。除日记展示外，进入 Room 时最近铭印会注入 system prompt（# Memory 块），Mobi 在对话中可引用 Star 教过的内容。
- **详见**：此前「故事驱动的自我学习与主动交互」计划（本文档施工表承接）。

## 四、Part 2: 日常节律

### 4.1 时段划分

| 时段 | 系统时间 | Mobi 状态 | 核心行为 |
|------|----------|-----------|----------|
| **早晨** | 6:00–12:00 | sleeping / waking | 用户戳/晃唤醒；Mobi 醒来后报梦（破冰话题） |
| **白天** | 12:00–22:00 | idle / parallelPlay | 用户沉默时 Mobi 做自己的事；偶尔抬头/挥手；发现困惑时主动问 |
| **晚上** | 22:00–6:00 | reflecting / sleeping | 房间变暗；Mobi 画画、展示涂鸦、说晚安、钻进被子、屏幕变暗 |

### 4.2 与现有实现衔接

| 设计 | 现有 | 需新增 |
|------|------|--------|
| 睡觉/唤醒 | 无 | Mobi 状态：sleeping；戳/晃唤醒；Wake-Up 时发梦境报告 |
| 平行陪伴 | seeking 主动说话 | idle 时 Mobi 做自己的事（玩积木/看云/巡逻）；偶尔 glance |
| 时间驱动 | 无 | 按 Calendar 当前小时判断 morning/day/evening；驱动不同行为 |
| 晚安流程 | 无 | 22 点后：房间变暗、Mobi 画画、说晚安、钻进被子、屏幕变暗 |

### 4.3 状态机（示意）

```
[morning]  → sleeping → (user poke/shake) → waking → dreamReport → idle
[day]      → idle ↔ parallelPlay ↔ (story trigger) → seeking
[evening]  → reflecting → drawJournal → goodnight → sleeping → screenDim
```

## 六、执行顺序与上下文边界

**建议执行顺序：** A1 → A2 → A3 → B1 → B2 → C1 → C2 → C3 → C4 → D1 → D2 → D3 → E1；D4、B3、C5 可后续迭代。

**单 Agent 上下文边界：** 每完成 1–2 个任务即更新本表「验收」列为 ✅，并 commit；下一 Agent 读本表从第一个未 ✅ 的任务继续。

**防失忆清单（每任务开始前）：**
1. 读本表「输入」列文档与文件。
2. 确认「依赖」任务已 ✅。
3. 完成后在本表对应行标注 ✅，并同步 [当前玩法说明-基于现有实现](当前玩法说明-基于现有实现.md)。
